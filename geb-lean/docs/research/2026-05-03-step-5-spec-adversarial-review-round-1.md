# Step 5 spec ‚Äî Adversarial review round 1

## Summary

The spec for step 5 (`kToER` structural-recursion functor and
`kToERFunctor`) is largely well-grounded against the landed step 2 / step
4 / `KMor1` / `ERMor1` / `KSimMor` / `ERMorNQuo` API. The architectural
arc ‚Äî atoms / `comp` / `raise` definitional, plus a dedicated simrec
helper using `simultaneousBoundedRec_interp_correct` plus
`majorize_by_componentBound` ‚Äî is sound. Spec's overall expected file
size and its Factoring-R decomposition look reasonable.

However, several concrete gaps must be closed before the
implementer-time experience matches the spec's "predominantly structural
plumbing" framing. There is one clear blocker (a type mismatch between
the dominance hypothesis produced by `kToER_simrec_dominates` and the
shape consumed by `simultaneousBoundedRec_interp_correct`), several
majors (mis-stated induction hypotheses, an `addK` arity / privacy
mismatch, missing `ih`-then-rewrite ordering, undefined `h_h` / `h_g` in
the simrec assembly signature, banned-word usages), and a handful of
minors (no explicit import list, citation gaps, `gcongr` availability
unverified).

Counts: 1 blocker, 7 majors, 9 minors.

## Blockers

### B1. Dominance hypothesis does not match `simultaneousBoundedRec_interp_correct`

`simultaneousBoundedRec_interp_correct` (step 2,
`GebLean/Utilities/ERSimultaneousBoundedRec.lean` line 405) requires its
`h_dominates` argument in the form

```text
‚àÄ (m : ‚Ñï), m ‚â§ n ‚Üí ‚àÄ (j : Fin (k + 1)),
  Nat.simRecVec k a (fun j' => (h j').interp)
                    (fun j' => (g j').interp) m x j
    ‚â§ componentBound.interp (Fin.cons m x)
```

where `h, g` are the families *passed to `simultaneousBoundedRec`*.
At the call-site in ¬ß6.1.4 the spec invokes
`ERMor1.simultaneousBoundedRec k a (fun j => kToER (h_fam j) _)
(fun j => kToER (g_fam j) _) bound i`. So `h, g` here are the
*kToER-translated* ER families, and the hypothesis demanded is

```text
Nat.simRecVec k a (fun j' => (kToER (h_fam j') _).interp)
                  (fun j' => (kToER (g_fam j') _).interp) m x j
  ‚â§ bound.interp (Fin.cons m x).
```

But `kToER_simrec_dominates` (¬ß6.1.2) produces the *K^sim-side*
dominance with families `(h_fam j').interp` / `(g_fam j').interp`, not
the kToER-translated ones. The spec's ¬ß6.1.4 passes
`kToER_simrec_dominates i h_fam g_fam h n x` directly to
`simultaneousBoundedRec_interp_correct` ‚Äî type error.

The IH-rewrite in ¬ß6.1.4 is performed *after*
`simultaneousBoundedRec_interp_correct` is invoked, but
`simultaneousBoundedRec_interp_correct` consumes `h_dominates` *before*
returning the equation. So the IH must be applied to the dominance
hypothesis itself, not just to the post-rewrite goal.

Fix options:

1. State `kToER_simrec_dominates` directly in the kToER-translated
   form, threading `ih_h`, `ih_g` through it as additional
   hypotheses.
2. Keep `kToER_simrec_dominates` K^sim-side and expose a separate
   adapter `kToER_simrec_dominates_translated` that re-binds the
   families using the IHs before passing into the bounded-rec
   theorem.
3. In ¬ß6.1.4, materialize the IH-rewrite of the family functions
   before rewriting via `simultaneousBoundedRec_interp_correct` ‚Äî
   so that the dominance hypothesis can be used at the original
   K^sim shape (because by then `simRecVec` will have its families
   replaced via `funext`).

Option 3 is the cleanest. The spec must specify which path is
taken; as written, the proof does not type-check.

## Majors

### M1. `kToER_interp_simrec`'s IH arguments use undefined names `h_h` / `h_g`

The ¬ß6.1.4 signature reads

```lean
(ih_h : ‚àÄ (j : Fin (k + 1)) (v' : Fin a ‚Üí ‚Ñï),
  (kToER (h_fam j) (h_h j)).interp v' = (h_fam j).interp v')
```

but `h_h` is not bound anywhere in the theorem's scope. It was a local
`have` inside `kToER`'s simrec branch (¬ß4.2 lines 231‚Äì240). At the
top-level of `kToER_interp_simrec`, that `have` is not in scope. The
correct shape uses proof irrelevance, e.g.

```lean
(ih_h : ‚àÄ (j : Fin (k + 1)) (h' : (h_fam j).level ‚â§ 2)
          (v' : Fin a ‚Üí ‚Ñï),
  (kToER (h_fam j) h').interp v' = (h_fam j).interp v')
```

or alternatively builds the `h_h` / `h_g` inside the theorem and
passes them in. As written, the spec does not type-check.

The same flaw appears at ¬ß5.3's `induction f with | simrec ... =>
exact kToER_interp_simrec ... (fun j v' => ih_h j v') ...`: the IH
generated by `induction f` does not match the spec's signature for
`kToER_interp_simrec`.

### M2. Spec mis-states `addK`'s arity and privacy

Spec ¬ß9.2 line 731 claims "Use the `addK : KMor1 1` simrec witness
from Phase-1 task #239". In the actual codebase
(`GebLeanTests/LawvereKSimInterp.lean` line 47 and
`GebLeanTests/LawvereKSimMajorization.lean` line 7), `addK : KMor1 2`
(arity 2 ‚Äî two natural-number inputs), and it is `private`. The Tier-2
test pseudocode `(kToER addK ...).interp ![3, 5]` only works because
arity is 2; the spec text contradicts the example. Worse, since
`addK` is `private`, it cannot be imported from the test module
`GebLeanTests/LawvereKSimER.lean`. The spec acknowledges the
fall-back ("If `addK` is not exported, the implementer constructs an
inline `addK`") but the framing of "available from Phase-1 task #239"
is misleading.

Fix: either de-`private` `addK` in
`GebLeanTests/LawvereKSimInterp.lean` (and re-export), or specify
that step-5 inlines its own `addK` definition. Also correct the
arity claim from `KMor1 1` to `KMor1 2`.

### M3. `KMor1.interp_simrec`'s extracted shape is not `Fin.tail v` / `v 0`

The landed `KMor1.interp_simrec`
(`GebLean/LawvereKSimInterp.lean` line 161) produces

```text
KMor1.simrecVec h g (fun j => ctx (Fin.succ j)) (ctx 0) i
```

NOT `KMor1.simrecVec h g (Fin.tail ctx) (ctx 0) i`. While
`Fin.tail ctx` is *definitionally* equal to `fun j => ctx (Fin.succ j)`
(per mathlib's `Fin.tail` definition), the spec's pseudocode at ¬ß6.1.2
and ¬ß6.1.4 relies on `simp only [Fin.tail_cons, Fin.cons_zero]` to
rewrite, which only fires when the goal mentions `Fin.tail`
syntactically. Step 4's existing simrec proof
(`LawvereKSimMajorization.lean` line 814) used a `change
KMor1.simrecVec h_fam g_fam (Fin.tail v) (v 0) i ‚â§ _` precisely to
fix the syntactic mismatch.

The spec's ¬ß6.2 third risk callout flags this but does not bake it
into the ¬ß6.1.2 / ¬ß6.1.4 pseudocode. As written, the
`simp only [KMor1.interp_simrec, Fin.tail_cons, Fin.cons_zero]`
chain at ¬ß6.1.2 line 419-420 will fail to fire because the post-
unfold form has `(fun j => ctx (Fin.succ j))`, not `Fin.tail ctx`.

Fix: add an explicit `change` step (matching step 4's pattern), or
use `show` to force the syntactic shape, or add a dedicated
`@[simp]` lemma `simrecVec_via_Fin_tail`.

### M4. `majorize_simrec_index_indep`'s `rfl` claim is fragile

Spec ¬ß6.1.1 claims the proof is `rfl`. The `KMor1.majorize` simrec
branch (line 632) binds its `match` index to `_`, so the returned
*tuple* is `(2, r_H + r_G + 2)` regardless of `i` or `j`. However,
both `r_H` and `r_G` are computed via `Fin.foldr` taking a level
proof argument that depends on `hyp` (= `h_i` on the LHS, `h_j` on
the RHS). The two proofs `hh j : (h_fam j).level ‚â§ 1` derived from
`h_i` vs `h_j` are propositionally equal (Lean Prop proof
irrelevance), so the values *should* be definitionally equal.

The risk: Lean's reduction may not propagate proof-irrelevance through
`Fin.foldr`'s body when the proofs are inside a `have` introduced by
`by`-tactic blocks. The spec lists a `simp only [KMor1.majorize]`
fallback. This is plausibly fine, but the spec should mark it as
"verify empirically; rfl preferred but fallback ready". It is not a
blocker, but it is a real risk that warrants marking.

### M5. `kToER`'s simrec `let` chain may not satisfy the equation compiler

Spec ¬ß6.2 risk callout #2 already flags this, but only as a callout.
The simrec branch's body has four sequential `let` bindings
followed by a `simultaneousBoundedRec` invocation referencing all
four. In some Lean configurations the equation compiler does not
elaborate this as straightforwardly structurally recursive on the
input `KMor1`; in particular it may try to unfold the `let` bindings
into the recursion measure.

The recommended fix in the callout ‚Äî pull the simrec body into a
helper `kToER_simrec_body` ‚Äî should be promoted from "if needed" to a
positively-recommended structuring. Otherwise the implementer hits
the equation compiler at task time without a planned fallback.

### M6. `sumCtxER_cons_le_of_le` proof body is `...` with no concrete tactic plan

Spec ¬ß6.1.3 leaves the body of `ERMor1.sumCtxER_cons_le_of_le` as
`...`. The spec's tactic suggestion is `gcongr` "if available, else
induction". There is no verification that `gcongr` is in scope for
`Nat`-valued `Fin.foldr` over `max` / `+`. Step 4's `vMax_cons`
analogue used direct `Fin.foldr_succ` unfolding plus `omega`. For
`sumCtxER`, `Fin.foldr` over `+` is monotone in each slot, but
`gcongr` may need lemma annotations to handle `Fin.foldr` directly.

Fix: write out a concrete proof skeleton using `Fin.foldr_succ` (or
the existing `interp_sumCtxER` equation) and a direct slot-bumping
argument. Match the style of step 4's `vMax_cons`.

### M7. `kToERFunctor` law `rfl` is overly optimistic given Quotient layers

Spec ¬ß8.3 claims both close by `rfl`, with a `Quotient.sound`-and-
compute fallback. Walking through `ùüô n` in `LawvereKSimDCat 2`:
`‚ü®KMorNQuo.id n, KMorNQuo.id_atDepth‚ü©` where `KMorNQuo.id_atDepth` is
`Quotient.mk _ {rep := KMorN.id n, ...}`. The `liftOn` over the
data-bearing depth_witness then drops to the `rep` field, applies
`kToERN`, and `Quotient.mk`s the result.

Even if `kToERN (KMorN.id n) _ = ERMorN.id n` is definitionally true,
the `Quotient.liftOn ... (Quotient.mk _ ...)` outer wrapper produces
something that requires the lift's beta-reduction to fire. Lean's
default reducibility settings often do *not* unfold `Quotient.liftOn`
on `Quotient.mk` automatically ‚Äî that requires a `rfl` after `simp
only [Quotient.lift_mk]` or similar.

The spec's "`rfl` works" assumption is more wishful than verified.
Fix: state the more realistic plan as "`simp only [Quotient.lift_mk,
kToERN, KMorN.id, ERMorN.id, kToER]` followed by `rfl`", with the
bare `rfl` as an opportunistic optimization.

The same concern applies more strongly to `map_comp`, where two
`liftOn`s have to combine at the outer level.

### m5. Spec uses `Quotient.exact` on `rec‚ÇÅ.rep_eq.trans rec‚ÇÇ.rep_eq.symm` indirectly

¬ß8.2's well-definedness body has the lines

```lean
have hrel : kMorNSetoid n m |>.r rec‚ÇÅ.rep rec‚ÇÇ.rep := by
  apply Quotient.exact
  rw [rec‚ÇÅ.rep_eq, rec‚ÇÇ.rep_eq]
```

After the `rw`, the goal is `Quotient.mk _ rec‚ÇÅ.rep = Quotient.mk _
rec‚ÇÇ.rep` rewritten to `f.hom = f.hom`, and then
`Quotient.exact` of that... but `Quotient.exact` consumes
`Quotient.mk _ x = Quotient.mk _ y`, not `q = q`. The `rw` rewrites
both occurrences of `rec_i.rep_eq`'s LHS (the `Quotient.mk _ rec_i.rep`)
to `f.hom`, leaving `f.hom = f.hom`, then `Quotient.exact`
expects the `Quotient.mk` form back.

The proof as written is back-to-front. The standard idiom is

```lean
have h_eq : Quotient.mk (kMorNSetoid n m) rec‚ÇÅ.rep
          = Quotient.mk (kMorNSetoid n m) rec‚ÇÇ.rep :=
  rec‚ÇÅ.rep_eq.trans rec‚ÇÇ.rep_eq.symm
exact Quotient.exact h_eq
```

Fix: rewrite the well-definedness pseudocode to chain `rep_eq` and
its symm without `rw`-ing into a tautology.

### m6. `kToER`'s level-discharge `have`s don't always reduce to `‚â§ 2`

Spec ¬ß4.2's simrec case constructs `h_h j` and `h_g j` at level `‚â§ 2`
by first deriving `‚â§ 1` and then `omega`-ing up. This is fine for the
proof but produces a less-canonical proof term. If
`KMor1.majorize` *also* takes a `‚â§ 2` proof and that proof is
required to be definitionally a particular shape (e.g., for the
later `rfl` in `majorize_simrec_index_indep`), the level-discharge
shape mismatch could trigger reduction failures.

Recommend: derive the `‚â§ 2` proof via the same `le_trans`-only chain
that step 4's `KMor1.majorize` uses, without `omega`. This is the
"recommended for consistency" alternative noted in ¬ß1.3, but the
spec frames it as cosmetic when in fact it has elaboration
implications.

### m7. The Tier-1 `raise` test at ¬ß9.1 mis-states the arity

```lean
example {f : KMor1 0} (h : (KMor1.raise f).level ‚â§ 2)
        (h' : f.level ‚â§ 2) :
    kToER (KMor1.raise f) h = kToER f h' := rfl
```

`KMor1 0` is the empty-arity case (no projections available). It is
inhabited (e.g., `KMor1.zero`), so the test is meaningful, but it is
an unusual choice. Recommend a more typical example, e.g., `f : KMor1
2` with a level-2 inhabitant (also tests that `kToER`'s reduction
ignores `raise`'s level boost).

### m8. ¬ß3 line 174 makes a conditional addition without a verification step

```text
ERMor1.sumCtxER_cons_le_of_le [conditional]
  -- Added only if ¬ß6.1.3's bound-monotonicity proof needs it.
```

The implementer-discretion language is fine, but the spec should
either commit to adding it or commit to a non-helper proof path.
"Conditional" here is a deferral with no decision criterion ‚Äî what
exactly does the implementer check at task time to decide? Better:
"Default plan: add it at task time. Skip if `gcongr` plus

### m9. ¬ß8.4 opportunistic helpers signature is incomplete

`kToERFunctor_map_quot`'s signature includes

from `h_lvl` automatically, or note the implementer-side
reconstruction path.

## Recommendations

For round 2 (before transitioning to writing-plans), the spec must:

1. **Resolve Blocker B1.** Pick option 1, 2, or 3 from the B1 fix
   list and rewrite ¬ß6.1.2 / ¬ß6.1.4 accordingly. This is the load-
   bearing correctness gap.

2. **Fix M1 (induction-hypothesis signature).** Re-state
   `kToER_interp_simrec`'s `ih_h` / `ih_g` arguments to bind the level
   proofs `h'` explicitly. Update ¬ß5.3 to match.

3. **Fix M2 (`addK` arity / privacy).** Either de-`private` `addK` or
   spec inline reconstruction; correct the arity claim.

4. **Promote the M3 / M5 / M6 / M7 risk callouts to commitments.**
   Each of these is currently soft-flagged but, on examination, is
   likely to bite at implementation time.

5. **Fix the spec-internal banned words (m1).**

The blocker B1 is the only path-changing defect. Majors M1 and M2 are
quick edits. The remaining majors and minors are largely
documentation / proof-strategy precision; the spec's overall shape and
its API-grounding are correct.

What is acceptable as-is: the architectural decomposition (Factoring R
into three named simrec helpers), the choice of
`sumCtxERPlusOffset (a + 1) p.2` (correctly justified in ¬ß4.3 against
the master-design pseudocode's `sumCtxER (a+1)`), the Tier 1 / Tier 2 /
Tier 3 test stratification, and the "rfl-or-fallback" position on
functor laws (modulo the over-optimism flagged at M7).
