## MINOR findings

### M1. `collapseER : ERMorN a 1` versus the 6.5b multi-output map

Location: spec §6.4 (`collapseER`, `collapseER_interp`) and §6.3
("the a-ary form ... per the `ERMorN a 1` shape 6.5 consumes").

Defect: the spec commits `collapseER (F …) : ERMorN a 1` (output arity
`1`) and states `collapseER_interp` as `(collapseER F).interp v = …`
the denotation of a single closed source term `F` of codomain sort `o`.
It asserts "6.5 consumes `ERMorN a 1`". The parent 6.5b consumer,
however, is `collapseFunctor : SynCatFO ⥤ LawvereERCat`, whose morphism
map must send `g : Γ ⟶ Δ` to a `LawvereERCat` morphism of codomain
arity `objLen Δ`, and whose adequacy bridge is quoted as
`collapseER_interp : (collapseER F).interp = collapseDenotation g` for
a whole morphism `g`. The parent's `collapseDenotation`
(`docs/superpowers/plans/2026-07-04-ramified-p6-soundness-subplan.md`,
task 6.5a) has type
`(Fin (objLen Γ) → ℕ) → (Fin (objLen Δ) → ℕ)`, i.e. `objLen Δ` outputs.
When `objLen Δ > 1`, the single-output `collapseER : ERMorN a 1` does
not directly serve the morphism map, and neither the spec nor the
parent records the decomposition-plus-tupling that reconciles them.

Evidence: spec §6.4 code block (`ERMorN a 1`) and §6.3 line "the a-ary
form (inputs packed per the `ERMorN a 1` shape 6.5 consumes)"; parent
plan §6.5a `collapseDenotation` signature and §6.5b text "morphism map
the 6.4 `collapseER` of the morphism's applicative translation" and
"`collapseER_interp`: the interp of the landed `ERMorN` equals
`collapseDenotation g`". `LawvereKSim.lean:65`
(`KMorN n m := Fin m → KMor1 n`) confirms codomain arity is a free
parameter on the K side; `LawvereERCat` morphisms are likewise
multi-output.

The atomic reading is coherent: `collapseER` is the builder for one
codomain-`o` source component (output arity `1`), and `collapseFunctor`
assembles a general `g : Γ ⟶ Δ` by applying `collapseER` to each of the
`objLen Δ` codomain-`o` components and tupling into
`ERMorN (objLen Γ) (objLen Δ)`. Under that reading `ERMorN a 1` is the
correct 6.4 deliverable shape. The spec does not state it, and its flat
claim that "6.5 consumes `ERMorN a 1`" over-describes: 6.5b consumes a
tuple of per-component `collapseER`s and needs a tupling lemma lifting
per-component `collapseER_interp` to the whole-morphism
`collapseDenotation g`.

Proposed resolution: add one sentence in §6.4 (or §3) recording that
`collapseER` is the atomic builder for a single codomain-`o` source
morphism, that `collapseFunctor` (6.5b) obtains the general morphism
map by tupling `objLen Δ` per-component `collapseER`s into
`ERMorN (objLen Γ) (objLen Δ)`, and that a tupling lemma lifts the
per-component `collapseER_interp` to the parent's whole-morphism
`collapseDenotation g`. This keeps the tupling layer in 6.5 scope while
preventing a plan from reading `collapseER` as the whole morphism map.

## NIT findings

### N1. Arrow notation for `RType.arrow` in code-shaped blocks

Location: spec §6.3 (`F : … (RType.omega τ ⟶ RType.o)`) and the §6.4
`collapseER`/`collapseER_interp` blocks that inherit it.

Defect: the repository has no `⟶` notation for `RType`; `⟶` is
CategoryTheory's `Quiver.Hom`. The corresponding repository type is
written `RType.arrow (RType.omega τ) RType.o`
(`GebLean/Ramified/Soundness/Normalization.lean:3104`,
`prop13_elementary`). A grep for a scoped `⟶`/`infixr` notation over
`RType` returns nothing. The design-doc blocks are illustrative
(elided with `…`), so this is harmless in prose, but the shorthand does
not correspond to a real notation and could mislead a plan writing
compilable signatures.

Evidence: `GebLean/Ramified/RType.lean:116`
(`def RType.arrow (a b : RType) : RType := …`); no `⟶` notation for
`RType` in `GebLean/Ramified/`. `prop13_elementary` uses
`RType.arrow (RType.omega τ) RType.o`.

Proposed resolution: either write `RType.arrow (RType.omega τ) RType.o`
in the code-shaped blocks, or add a one-line note that `⟶` abbreviates
`RType.arrow` in this document. (The parent plan uses the same
shorthand; aligning both is optional.)

### N2. Value-laden superlative "bulkiest"

Location: spec §6.1 ("is the bulkiest proof obligation of the
sub-phase") and §8.2 ("the bulkiest arm-by-arm mirror").

Defect: `CLAUDE.md` § Style guidelines and
`.claude/rules/markdown-writing.md` § Prose style discourage
value-laden adjectives. "bulkiest" is an informal superlative;
"largest" is the neutral form, and the same content is already carried
by §8.2's "difficulty concentration".

Proposed resolution: replace with "largest" or drop; §8.2 already flags
the two concentrations.

### N3. Colloquial "hard spots"

Location: spec §8.2 ("Two flagged hard spots").

Defect: "hard spots" is a colloquialism; the style rules require
standard technical terms. §8.2's own heading "Difficulty concentration"
supplies the neutral phrasing.

Proposed resolution: "Two flagged difficulty concentrations" or "Two
declarations of concentrated difficulty".

- Section 4.1 count argument. `RType.ord (arrow σ τ) = max (σ.ord + 1)
  τ.ord ≥ σ.ord + 1 > σ.ord` (`Normalization.lean:143-157`), so
  `ord σ < ord (σ → τ)` holds; claim (i) is correct. The
  substitution-site bound is exactly `betaRedexRank_instantiate₁_le`
  (rank `≤ RType.ord σ` on the freshly created redexes). Innermost
  selection (child-priority descent, contracting only when no child
  carries a rank-`q` redex) makes `D` and `E` free of rank-`q` redexes,
  so duplication of `E` introduces no rank-`q` redex; claim (ii) is
  correct. The descent rule is total: `betaRedexRank t = q` forces
  `topBetaRank t = q` or a child with `betaRedexRank = q`
  (`redexRank_app'` decomposition), so no node is left undischarged.
- β/ι interleaving. `exists_iota_step_of_hasIota`
  (`Normalization.lean:1786`) returns a reduct with
  `betaRedexRank t' = 0`: an ι-contraction on a β-normal term preserves
  β-normality and cannot create a β-redex. `lemma12`
  (`Normalization.lean:2656`) is `beta_normalize` then `iota_normalize`
  with no loop-back. The spec's `detStep` selection rule ("β while
  `betaRedexRank > 0`, else ι") mirrors this two-phase order exactly;
  the "else if `hasIota`" phase never returns to β, so `detIter_normal`
  re-deriving the `lemma12_clock` bound is plausible. A β-contraction
  may create ι-redexes, but those are consumed in the later ι-phase, so
  the phase split is not mishandled.
- η. `OneLambdaStep` includes an `eta` rule
  (`OneLambda.lean:205-210`), but `redexRank`/`Normal`
  (`Normalization.lean:694-702`) have no η component:
  `Normal = betaRedexRank = 0 ∧ hasIota = false`. `Normal` is therefore
  β/ι-normal, consistent with the spec §1.1 non-goal; `detStep`
  performing only β and ι contractions is consistent with
  `detStep_sound : ¬ Normal t → OneLambdaStep t (detStep t)`, since a
  non-`Normal` term has a β or ι redex.
- §5.4 sort stability. `instantiate₁` substitutes existing subterms and
  the `dstrHit`/`dstrMiss`/`case` reducts select existing subterms
  (`OneLambdaStep`, `OneLambda.lean:211-244`); neither introduces a sort
  absent from the source, so the "β and ι contraction introduce no sort
  absent from the source term" claim holds. η is excluded from `detStep`
  and is irrelevant here.
