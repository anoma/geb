## Decisions recorded by this mini-sub-plan (for adversarial + user review)

### Decision 1 — `SimpleTy` is the omega-free fragment of `RType`

`1λ(A)`'s types are the simple types over base `o`. Realize them as `RType`
restricted to the `{o, arrow}` fragment: `oneLambdaSig` is a `BinderSig RType`
whose operations only ever produce `o`/`arrow` sorts, and a new predicate
`RType.IsSimple` (omega-free, mirroring `RType.IsTower`, ~10 lines +
`DecidablePred`) is the faithfulness invariant proving the bar-image lands in
the fragment. A fresh two-shape `PolyFix`/`FreeAlg` for a separate `SimpleTy`
would duplicate `FreeAlg`/`DecidableEq`/`RType.curried`/`RType.interp` (which
already sends `o ↦ carrier`, `arrow ↦ →` and never sees `omega` on the pure
fragment) for marginal purity — a "Code is cost" loss. Keeping both calculi
over one `RType` universe lets `barTm` be a plain
`Tm (rlmrOSig …) → Tm (oneLambdaSig …)` over the same de Bruijn machinery.

### Decision 2 — `Represents` anchors the source value denotationally (deviation)

Leivant's `Represents` (§4.2) is operational on both sides: at object type,
`F` reduces (in `RλMR_o^ω`) to some `a`, and `F̂` represents `F` iff `F̂`
reduces (in `1λ(A)`) to `a`. The source-side "reduces to `a`" is Lemma 9 —
canonicity of the source calculus, which in general requires strong
normalization of a System-T-like calculus (a large reducibility-candidates
proof, far outside this scope).

This sub-plan replaces the source-side operational value by the denotational
value `appEval F` (6.1's total standard-model evaluator, proven to equal
`RIdent.interp` on translations):

- Object type `o`: `Represents o F F̂ := ReflTransGen OneLambdaStep F̂
  (conc (appEval F Fin.elim0))`, where `conc : FreeAlg natAlgSig → Tm
  (oneLambdaSig natAlgSig) [] o` is the concrete constructor term of a value and
  `OneLambdaStep` is the CONGRUENCE-CLOSED target step (decision 4).
- Ω-type `Ωτ`: `Represents (Ωτ) F F̂ := ReflTransGen OneLambdaStep F̂
  (bbRep (appEval F Fin.elim0) τ̄)`, the abstract Berarducci–Böhm representation
  of the value at `τ̄`.
- Arrow `σ→τ`: `Represents (σ→τ) F F̂ := ∀ G Ĝ, Represents σ G Ĝ →
  Represents τ (sourceApp F G) (app' F̂ Ĝ)` (the logical-relation clause,
  unchanged from Leivant).

**Justification (mark in the code):** (a) 6.3's Lemma 12 bounds the number of
`1λ(A)` reduction steps — the TARGET side, which stays operational here; the
source-side reduction is never measured, so anchoring it denotationally costs
the soundness result nothing. (b) `appEval` is total and computable and
`prop7Translate_interp` already ties it to `RIdent.interp`, so Lemma 9 becomes
trivial (`a := appEval F Fin.elim0`) with no source normalization. (c) The
composite with 6.1 reads: `barTm (prop7Translate d)` reduces in `1λ(A)` to
`conc (RIdent.interp d)` — exactly what 6.4's normalizer consumes. (d) EXTRA
BENEFIT (adversarial review): because `Represents` reads the source only through
`appEval`, the source step `RlmrOStep` is never invoked, so ALL reduction
metatheory (congruence closure, confluence) is localized to the TARGET
`OneLambdaStep` alone — the source relation stays as 6.1 left it.

**Fidelity ledger:** this is NOVEL PACKAGING, not transcription. Leivant's
`Represents`/Lemma 9/Prop 11 are operational bi-conditionals (source reduces to
`a` and target reduces to `a`); this sub-plan's are denotational reformulations
of the same extension (they coincide via the avoided source canonicity). Tasks
6.2.6, 6.2.8, 6.2.10 docstrings mark them as deviation (Leivant §4.2–4.3, novel
packaging), NOT verbatim transcription. **The user should scrutinize this
deviation** — the fallback is Leivant's fully-operational `Represents`, which
would add a source-calculus strong-normalization sub-project (comparable in size
to all of 6.1).

### Decision 4 — the target step `OneLambdaStep` is congruence-closed

`Represents` (object clause) is `ReflTransGen OneLambdaStep F̂ (value)`, and
Prop 11 must inhabit it in cases whose reductions occur UNDER application spines
(case 2 `c_i M̂_1…M̂_{r_i}` has no top-level redex — the head is a constructor;
cases 5 and 7 reduce a proper subterm). So `OneLambdaStep` is NOT top-level-only
(unlike 6.1's `RlmrOStep`, which was never actually used in a reduction chain):
it is a compatible/congruence closure — the base redex rules (β/η/dstr/case)
PLUS structural constructors reducing under `app'` (left and right) and under
`lam'` (body). Task 6.2.2 budgets the congruence constructors and the
`ReflTransGen`-congruence helper lemmas (`app'`/spine compatibility) that cases
2/5/7 consume. Per decision 2(d) this is needed on the TARGET step only.
