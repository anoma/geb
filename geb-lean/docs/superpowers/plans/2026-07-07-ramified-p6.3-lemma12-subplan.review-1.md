### Adjudication of the mathematical checks

**N3, walked in full.** The invariant (4) (`IsLam t' → IsLam t ∨
ord s ≤ q`) is exactly strong enough. Application case `t = app' D E`
with `D : σ ⟶ τ`: if `D'` is an abstraction, then either `IsLam D`
(so the top redex was in `t` and `ht : betaRedexRank t ≤ q` gives
`ord (σ ⟶ τ) ≤ q`) or invariant (4) on `D` gives `ord (σ ⟶ τ) ≤ q`
directly; combined with the firing condition `ord (σ ⟶ τ) ≥ q` this
forces equality, whence `ord σ + 1 ≤ q` and N2 caps the contractum's
rank at `q - 1`. In the non-firing subcase a surviving top redex
forces `IsLam D'`, so non-firing means `ord (σ ⟶ τ) < q`, giving
rank `≤ q - 1` without further case analysis. Re-establishment of
(4) after contraction: `instantiate₁ E' b` may be an abstraction
while `t` is an application, and the stated disjunct
`ord τ ≤ ord (σ ⟶ τ) = q` covers it; all other cases are vacuous
(`t' = t`, `t'` an application, or `lam'`-to-`lam'`). Height:
`height b + height E' ≤ (2^(h-1) - 1) + 2^(h-1) < 2^h`; the
non-contraction and abstraction cases give `1 + 2^(h-1) ≤ 2^h`
(`h ≥ 1` by positivity). Steps: `k_D + k_E + 1 ≤ size t`.

**N2 coverage.** New β-redexes arise only at an application node
whose function child was the substituted variable: a substituted
term under further applications sits below an `app`-headed (not
`lam`-headed) function child, so intrinsic typing admits no redex
there, and argument-position occurrences create none. The statement
is correct. The sketched auxiliary invariant is not (R1-M2).

**N6.** `dstrHit`: contractum `a ⟨j, h⟩`, a proper subterm of the
scrutinee — strict. `dstrMiss`: contractum is the scrutinee itself;
the redex adds the `app` node and the nullary `dstr` constant, so
the size decreases by exactly 2 — strict. `case`: contractum
`b idx`, a proper constituent of the spine — strict. All contracta
have sort `o` and are placed at sort-`o` positions; intrinsic typing
forbids a sort-`o` term in function position, so no new β-redex
forms at the junction, and β-normality is preserved. New ι-redexes
may form (a `dstrHit` contractum is `con`-headed and may meet an
enclosing `dstr`), which the strong induction on size absorbs.

**N4/N5 arithmetic.** Cycle `j` starts at height `≤ tower (j-1) h`,
so its step count is `≤ size ≤ 2^(tower (j-1) h) = tower j h`;
`Σ_{j=1..q₀} tower j h ≤ q₀ · tower q₀ h` by `tower_mono_left`.
Per-cycle `M_j = size t_j + 2^(2^(height t_j)) ≤ tower j h +
tower (j+1) h ≤ 2 · tower (q₀+1) h ≤ tower (q₀+1) (h+1)` using
`tower k x + 1 ≤ tower k (x+1)`. The `hM` invariant of `beta_cycle`
is inductively stable: lifted intermediates `app' D_mid E` and
`app' D' E_mid` are bounded by `size t + 2^(2^(h-1)) + 2^(2^(h-1))
≤ size t + 2^(2^h)`, and the substitution endpoint by
`2^(2^(h-1) + 2^(h-1) - 1) ≤ 2^(2^h)`.

**P4 and Task 6.3.12.** `lemma8` (`Represents.lean:175`) prepends
reduction to a representative (`Fhat' ⇒* Fhat` transfers
representation to `Fhat'`); it cannot carry representation forward
to the `lemma12` reduct, so the evaluator route is necessary as
claimed. The assembly is logically complete:
`prop11_represents F` + `represents_arrow` (an `Iff.rfl` unfold) +
`lemma9_omega τ (sourceWord a τ)` rewritten by `appEval_sourceWord`
yield the sort-`o` clause
`W_a ⇒* conc (appEval (sourceApp F (sourceWord a τ)) finZeroElim)`;
`lemma12` yields the bounded chain to a `Normal` `n`; Task 6.3.10
gives `n = conc b`; `oneEval_reduces` along both chains plus
`oneEval_conc` and `conc_injective` force `conc b` equal to the
value word, so the bounded chain ends at the target. Elaboration
precedent for `app' (barTm F) (bbRep a (barTy τ))` and for
`bbRep … : Tm _ [] (barTy (Ω τ))` exists in the landed statements
of `represents_arrow` and `lemma9_omega`. `bbRep`'s argument order
(`a` then sort) matches `BarRep.lean:183`.

## Findings

### Major

#### R1-M1 — Task 6.3.9a's `oneEval` environment type contradicts its pin and test

Location: Task 6.3.9a, Interfaces produced; Step 1.

The plan states

```lean
(env : ∀ (u : RType) (x : Binding.Var Γ u), RType.interp (FreeAlg natAlgSig) u)
```

but the pattern it pins, `appEval` (`Applicative.lean:518-520`),
takes

```lean
∀ i : Fin Γ.length, RType.interp (FreeAlg natAlgSig) (Γ.get i)
```

and the task's own Step-1 test `oneEval (conc a₀) finZeroElim = a₀`
elaborates only under the `Fin`-indexed convention (`finZeroElim`
has type `∀ i : Fin 0, C i`, not `∀ u, Var [] u → …`). The fusion
lemmas the task mirrors (`appEval_ren`/`appEval_sub`) are likewise
stated `Fin`-indexed. The "conventions pinned at implementation"
hedge does not license two mutually inconsistent explicit forms
inside one task contract.

Suggested fix: state `oneEval`'s environment as
`∀ i : Fin Γ.length, RType.interp (FreeAlg natAlgSig) (Γ.get i)`,
matching `appEval` verbatim.

#### R1-M2 — N2's auxiliary environment invariant is not stable under `Env.underBinder`

Location: Mathematical notes N2; Task 6.3.5, interfaces prose and
Step 4.

The sketch generalizes to "all environment images' ranks and sorts
bounded" and asserts stability because "the new image is a variable
(rank 0, sort recorded)". The rank bound is stable, but the sort
bound is not: the freshly bound sort of a `lam` node is arbitrary,
so an invariant bounding `ord` of every image's sort by
`max (… , ord σ)` fails at the binder case. The correct invariant
exempts variable images from the sort bound (a variable in function
position forms no β-redex; only non-variable images need
`betaRedexRank ≤ K ∧ ord (sort) ≤ K'`). The final statement
`betaRedexRank_instantiate₁_le` is unaffected (the instantiating
environment maps all but the last variable to variables), but the
note as written misdirects the implementer of the step the plan
itself labels "the hard part".

Suggested fix: restate N2's auxiliary as: every image is either a
variable or has β-redex rank and sort order bounded; note that
`Env.underBinder`'s fresh image satisfies the variable disjunct.

### Minor

#### R1-m1 — N5's justifying sentence does not bound the mid-chain hybrids

Location: Mathematical notes N5.

"Its size is at most `Tm.size t + Tm.size t'` for the cycle's
endpoints" is not the operative bound: in the contraction case the
intermediate `app' D' E_mid` can exceed `size t + size t'` when the
substitution shrinks (`size t' = size ([E'/x]b)` may be smaller than
`size D'`). The bound that closes — and that Task 6.3.6's `hM`
actually states — is `size t + 2^(2^(Tm.height t))` (verified in
the adjudication above). The Lean proof is the checker of record and
the plan permits adjusting `M`, but the note should not record an
invariant that fails.

Suggested fix: restate N5's per-cycle claim as
`size ≤ Tm.size t + 2 ^ (2 ^ Tm.height t)` (the `hM` expression),
keeping the cross-cycle absorption into `S q₀ h` as written.

#### R1-m2 — `size_le_pow_height`'s hint chain does not close

Location: Task 6.3.2, Step 4.

The hint `1 + Σ ≤ 1 + m * max ≤ m ^ (1 + max)` is off by one:
with children bounded by `m ^ h_j` it yields `1 + m ^ height`,
which exceeds `m ^ height`. The statement is true; the standard
proof strengthens the invariant to `size ≤ m ^ height - 1`
(nullary: `1 ≤ m - 1` for `m ≥ 2`; node:
`1 + m · (m^(h-1) - 1) = m^h - m + 1 ≤ m^h - 1`), from which the
stated form follows.

Suggested fix: replace the Step-4 hint with the `- 1`-strengthened
induction.

#### R1-m3 — no task produces `app'`/`lam'` measure equations or transport invariance

Location: Task 6.3.1 interfaces; Tasks 6.3.5, 6.3.6, 6.3.12
(consumers).

`OneLambda.app'` transports its children along
`(List.append_nil Γ).symm ▸` (`OneLambda.lean`, whose docstring
records that `Γ ++ [] = Γ` is not definitional), so the generic
`size_op`/`height_op` equations do not directly yield
`height (app' f x) = 1 + max (height f) (height x)` or the `size`
analogue. These node-level equations are consumed throughout (the
congruence lifts' "exact additive constants from `size_op`", the
`beta_cycle` bookkeeping, Task 6.3.12's height bound on `W_a`), yet
no interface list contains them or the kit-level transport lemma
(`size (h ▸ t) = size t` for a context equality, provable by
`subst`) they rest on.

Suggested fix: add context-transport invariance of `size`/`height`
to Task 6.3.1 (kit-level, next to `size_ren`), and
`size_app'`/`height_app'`/`size_lam'`/`height_lam'` simp equations
to Task 6.3.4a's or 6.3.5's interfaces.

Suggested fix: a name in the `succ_tower_le_tower_succ` or
`tower_add_one_le_tower_succ_right` pattern, fixed at
implementation; place it in `Utilities/Tower.lean` (it is generic

`stlcId`. The "(names per `Examples/Stlc.lean`)" hedge covers it,
but the illustration should use the real names.
