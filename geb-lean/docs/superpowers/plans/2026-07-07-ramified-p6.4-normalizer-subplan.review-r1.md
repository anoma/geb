## MINOR findings

### M1. Course-of-values precedent mischaracterized

Location: plan §"Mathematical notes" N4 (line 343-351), Decision P3
(lines 132-142), "Consumed interfaces" (line 232), "Tech Stack"
(line 72), and Task 6.4.12 Step 1 (lines 916-920).

Defect: the plan repeatedly anchors the ER realization of strong
recursion on codes to `treeFoldOnCode`/`mutuFoldOnCode`
(`GebLean/LawvereTreeERArith.lean:23,149`), describing them as "an
`ERMor1.boundedRec` whose state packs the table of values at all smaller
codes (Szudzik sequence packing, `seqPack`/`seqAt` shapes)" (N4), and
Decision P3 asserts "strong recursion on codes is the shape those
precedents already realize". Both descriptions are inaccurate:

1. `treeFoldOnCode` and `mutuFoldOnCode` are declared in the
   **`TreeERMor1`** category, not `ERMor1`. Their signatures return
   `TreeERMor1_1 1` and they are built from `TreeERMor1_1.fold1` /
   `TreeERMor1_1.fold` over a `BT` (binary-tree) carrier with
   `decodeBT`/`encodeBT` round-tripping — a tree-fold combinator, not an
   `ERMor1.boundedRec` with a packed history table. No `TreeERMor1 →
   ERMor1` lowering appears in `LawvereTreeERArith.lean` (a grep of the
   file's `def`s produces only `TreeERMor1_*` outputs). The plan's
   deliverables are `ERMor1` (`normStep : ERMor1 1`,
   `normRun : ERMor1 2`), so the cited precedent is in a different
   morphism category.
2. The genuine `ERMor1`-level course-of-values-on-Gödel-code combinator
   is `ERMor1.foldBTLOnCode`
   (`GebLean/Utilities/ERTreeArith.lean:607`), whose module docstring
   states it "builds on `ERMor1.boundedRec` plus a Gödel β-encoded fold
   table" (ERTreeArith.lean:11). Its bound discharge is exactly the
   shape the plan wants (`boundedSearch` over `boundedRecRange bound`,
   an outer `minN` for unconditional boundedness, correctness
   conditional on `bound` dominating the fold trace —
   `ERMor1.interp_foldBTLOnCode_le_bound`,
   `ERMor1.foldBTLPred_trace_match`). The plan never cites it.
3. The table mechanism the actual `ERMor1` precedent uses is **Gödel
   β-encoding** (`Nat.foldBTLOnCode` via `ERMor1.beta`), not the "Szudzik
   sequence packing, `seqPack`/`seqAt` shapes" N4 names. `Nat.seqPack` /
   `Nat.seqAt` do exist (`GebLean/Utilities/SzudzikSeq`, used in
   `GebLean/LawvereKSimPolynomialBound.lean:713-867`), but in the K-sim
   polynomial-bound context, not as the packing of an `ERMor1`
   course-of-values fold.

Evidence: `LawvereTreeERArith.lean:23-29,149-154`
(`TreeERMor1_1`/`TreeERMor1_0`, `fold1`/`fold`);
`ERTreeArith.lean:11,607-640`
(`ERMor1.foldBTLOnCode`, "`boundedRec` plus a Gödel β-encoded fold
table"); plan lines 72, 137, 232, 343-351, 916-918 (no occurrence of
`ERTreeArith`, `foldBTLOnCode`, `BTL`, or "β-encod" in the plan).

Impact: `normStep` (Task 6.4.12) is one of the three flagged difficulty
concentrations, and its Step-1 re-checkpoint sends the implementer to
`LawvereTreeERArith.lean`/`LawvereNatBTV2.lean` — the wrong files —
risking a from-scratch `boundedRec`+`seqPack` build that duplicates the
existing `foldBTLOnCode`, against CLAUDE.md § "Reuse existing process
code" / "Reuse existing abstractions". The re-checkpoint is a partial

that does not lower to `ERMor1`.

Proposed resolution: re-point the course-of-values citations (N4, P3,
"Consumed interfaces", "Tech Stack", Task 6.4.12 Step 1) to
`ERMor1.foldBTLOnCode` (`GebLean/Utilities/ERTreeArith.lean:607`) and its
`boundedRec`-plus-β-encoded-table mechanism, and correct the packing
description from "Szudzik seqPack/seqAt" to the Gödel β-encoded fold
table. Retain `treeFoldOnCode`/`mutuFoldOnCode` only as a secondary
illustration if a `TreeERMor1 → ERMor1` lowering is confirmed to exist
(none was found).

### M2. N2 payload-positivity is false; `pair_one_lt` hypothesis superfluous

Location: plan §"Mathematical notes" N2 (lines 316-326) and Task 6.4.6
(`pair_one_lt (p : ℕ) (hp : 1 ≤ p) : p < Nat.pair 1 p`, line 642, plus
the strictness-cluster prose lines 648-656).

Defect: N2 states "Op-node codes are `Nat.pair 1 payload` with
`payload ≥ 1`; `Nat.pair 1 p > p` for all `p ≥ 1` (case split on
`1 < p`)", and Task 6.4.6 gives `pair_one_lt` the hypothesis `1 ≤ p`.
The `payload ≥ 1` claim is not established for the raw encoding: with
`codeRType RType.o = Nat.pair 0 0 = 0`, an `app`-node payload
`Nat.pair (codeOp o) childrenPack` can be `0` (it vanishes exactly when
`codeOp = 0` and `childrenPack = 0`, e.g. `codeOp (app o o) = 0` with
zero-coded children). Whether such a node is inhabited by a well-typed
term is a typing question the code level cannot see; the strong-recursion
functions (`subCode`, `stepCode`, `ordCode`, the detectors) are
`ℕ → ℕ` and their well-foundedness is a statement about the `Nat.pair`
encoding, not restricted to `codeTm` images. The termination measure the
recursion actually needs is `child code < op-node code`, which holds
unconditionally: `Nat.pair 1 p > p` for **all** `p` (at `p = 0`,
`Nat.pair 1 0 = 1·1+1+0 = 2 > 0`), and each child code `≤ payload` by
chained `left_le_pair`/`right_le_pair`. So the `1 ≤ p` hypothesis on
`pair_one_lt` is unsatisfiable precisely at the payload-`0` nodes the
recursion must still handle, while the unconditional statement discharges
them.

Evidence: mathlib `Nat.pair a b = if a < b then b*b+a else a*a+a+b`
(`.lake/packages/mathlib/Mathlib/Data/Nat/Pairing.lean:38-40`);
`Nat.left_le_pair`/`Nat.right_le_pair`
(ibid. lines 102, 104); `codeRType RType.o = Nat.pair 0 0`
(plan line 591). Compute `Nat.pair 1 0 = 2`, `Nat.pair 1 p = p·p+1` for
`p ≥ 2`, `Nat.pair 1 1 = 3`, so `p < Nat.pair 1 p` for every `p` with no
hypothesis.

Impact: low but real — an implementer following N2 may attempt to prove
the false `payload ≥ 1` lemma or find `pair_one_lt`'s hypothesis
unsatisfiable at a node it must recurse through. The strictness cluster
carries implementer latitude, so the fix is local.

Proposed resolution: state the strict step as `∀ p, p < Nat.pair 1 p`
(drop the `1 ≤ p` hypothesis) and remove the "payload ≥ 1" claim from
N2; the well-foundedness the recursion consumes holds unconditionally.

### N1. Decision P4 "exactly the two sides of `prop13_elementary`"

Location: plan Decision P4 (lines 144-154).

Defect: P4 says the adequacy right-hand side
`freeAlgToNat (appEval (sourceApp F (sourceWord a τ)) finZeroElim)` is
"exactly the two sides of `prop13_elementary` (`Normalization.lean:3103-3110`)".

is correct; the phrase "exactly the two sides" is loose. Harmless, but a
one-word qualifier ("the applied term and the word inside its `conc`
normal form") would prevent a plan reader from expecting the RHS spellings
to coincide literally.

`conc (appEval (sourceApp F (sourceWord a τ)) finZeroElim)`);
